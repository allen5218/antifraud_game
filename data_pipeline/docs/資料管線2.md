  優先用規則、SQL、Python、正規表示式、來源 parser

## 你的流程中最需要注意的 4 個坑

### 1. 「案件發生時間」不一定能準確抽出

新聞常常只有：

```text
近日
今年初
去年 8 月
警方日前接獲報案

所以 schema 裡不要只放：

"case_happened_at": "2026-06-01"

建議加：

"case_time_text": "去年 8 月",
"case_happened_at": null,
"case_time_precision": "month",
"case_time_confidence": 0.6

否則 Codex / LLM 可能會硬猜日期。

2. 「詐騙手法」要有受控詞表

不要讓資料裡出現一堆同義詞：

假投資
投資詐騙
虛假投資
投資型詐騙
股票群組詐騙
LINE 投資詐騙

建議定義：

scam_methods:
  - investment_scam
  - romance_scam
  - fake_police_scam
  - phishing_sms
  - job_scam
  - crypto_scam
  - mule_account

然後 JSON 可以同時保存：

{
  "scam_methods": ["investment_scam", "crypto_scam"],
  "scam_method_text": ["假投資", "虛擬貨幣投資詐騙"]
}

這樣關聯式表格比較乾淨。

3. JSON Schema 只能驗格式，不能驗內容真偽

JSON Schema 可以檢查：

欄位有沒有存在
型別對不對
日期格式對不對
URL 格式對不對
enum 有沒有符合

但不能保證：

案件時間是不是真的
詐騙手法判斷是否正確
標題是否完整
正文是否漏抓

所以你還需要：

validator
quality_check
sampling review
pipeline auditor
4. pgvector 不要直接吃 JSONB

pgvector 應該吃的是乾淨的 chunk：

documents.body_text
documents.summary
documents.prevention_tips
judgment facts

不要直接把整包 JSON stringify 後 embedding。
那樣會把欄位名稱、雜訊、爬蟲 metadata 都混進向量。
