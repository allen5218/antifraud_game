Thought for 10s

可以，但不應該把它當成穩定的「skill 互相引用機制」。

目前官方文件明確說的是：

Codex 啟動時只會把每個 skill 的 name、description、file path 放進上下文。
當任務匹配某個 skill，或使用者用 $skill-name / /skills 明確指定時，Codex 才會讀取該 skill 的完整 SKILL.md。
skill 的標準結構是 SKILL.md 加上自己的 scripts/、references/、assets/，也就是設計上偏向「一個 skill 內部引用自己的參考檔」，不是「skill A 正式 import skill B」。

所以你問的情況：

在 skill-A 的提示詞中寫：
完成這一步後，請使用 $skill-B

理論上可能有效，因為 Codex 的初始上下文知道可用 skill 的名稱與描述，而且官方也說 CLI/IDE 可以用 $ 明確 mention skill。
但它更像是「自然語言指示 Codex 去啟用另一個 skill」，不是規格保證的 dependency / import / include。

比較可靠的做法是：

---
name: scam-crawler
description: Use this skill when crawling scam case websites and extracting structured scam reports.
---

# Workflow

1. Before crawling, inspect existing records in PostgreSQL.
2. For extraction rules, read `references/schema.md`.
3. For Playwright crawling procedure, read `references/playwright-crawling.md`.
4. For validation, run `scripts/validate_json.py`.

也就是把共用流程拆到 references/ 或 scripts/，而不是期待它自動讀另一個 skill。

如果你真的想要 modular skill，可以這樣寫，但要視為提示技巧：

---
name: ecommerce-etl
description: Use this skill when importing ecommerce data into PostgreSQL JSONB and normalizing it.
---

# Workflow

1. Use this skill as the coordinator.
2. If the task requires browser crawling, invoke `$playwright-crawler`.
3. If the task requires JSON schema validation, invoke `$json-schema-validator`.
4. If the task requires PostgreSQL import, invoke `$postgres-jsonb-importer`.
5. After each delegated step, return to this workflow and continue.

我會建議你把它分成兩層：

穩定方案：
一個 orchestration skill，裡面用 references/ 寫清楚整套流程；共用腳本放 scripts/。這符合 Agent Skills 的 progressive disclosure 設計：先讀 metadata，需要時再讀 full SKILL.md，再按需讀 references/resources。

實驗方案：
在 orchestration skill 裡點名 $other-skill，讓 Codex 嘗試載入另一個 skill。這可能能跑，但容易受版本、skill list 是否被截斷、名稱衝突、描述是否被裁切影響。官方文件也提到，skill 列表最多只用 2% context 或 8000 字元；skills 太多時，描述可能被縮短，甚至有些 skill 會從初始列表省略。

結論：Codex CLI 支援在使用者 prompt 中用 $skill-name 明確叫 skill；但「skill 內寫另一個 skill 名稱讓 Codex 自動讀取」目前更像 prompt chaining，不是正式依賴機制。 做可維護流程時，最好用 references/、scripts/、或把多個相關 skills 包成 plugin；官方也說 plugin 可以包含一個或多個 skills。

英文可以這樣問：

Does Codex CLI support invoking one skill from inside another skill by mentioning the other skill’s name?

或更精確：

Can a Codex skill trigger another skill via $skill-name, or should shared logic be placed in references and scripts instead?

如果我要做一個資料管線的skills，要使用playwright-cli，應該把playwright-cli的skills合併進這個skills，變成references/嗎

我會把它拆成「可維護性」和「Codex 實際觸發可靠性」來判斷；這題的關鍵不是能不能合併，而是管線 skill 是否能穩定把 Playwright 步驟跑完。

Thought for 6s

是，我會建議合併進你的資料管線 skill，放成 references/，但不要整包照搬成另一個完整 skill。

比較好的結構是這樣：

data-pipeline/
├── SKILL.md
├── references/
│   ├── playwright-cli.md
│   ├── extraction-schema.md
│   ├── crawling-policy.md
│   ├── postgres-jsonb.md
│   └── normalization.md
├── scripts/
│   ├── validate_json.py
│   ├── normalize_jsonb.py
│   └── embed_to_pgvector.py
└── examples/
    ├── sample_output.json
    └── sample_urls.txt

原因是：官方對 Codex skills 的定位是「一個 skill 包裝一套任務流程，包含 instructions、resources、optional scripts」，而 references/ 很適合放 Codex 需要按需讀取的細節文件，例如 schema、API 文件、資料庫結構、詳細流程指南。 Playwright 官方也把 playwright-cli 定位成給 coding agents 用的 CLI browser automation，重點是用精簡 CLI 指令控制瀏覽器，避免 MCP 那種大 schema / accessibility tree 長期佔 context。

你的資料管線 skill 應該是 orchestrator skill，Playwright CLI 只是其中一段「網頁擷取能力」。所以不要做成：

data-pipeline/
├── SKILL.md
└── skills/
    └── playwright-cli/
        └── SKILL.md

也不要主要依賴：

請去使用 $playwright-cli skill

這種做法比較像 prompt chaining，不是正式的 dependency/import 機制。比較穩的是把 Playwright CLI 的必要指令、流程、注意事項、輸出格式整理成 references/playwright-cli.md，讓主 SKILL.md 在需要爬取時明確要求讀它。

你的 SKILL.md 可以長這樣：

---
name: data-pipeline
description: Use this skill when crawling target websites, extracting structured records, validating JSON, storing raw JSONB in PostgreSQL, normalizing relational tables, and generating pgvector embeddings.
---

# Data Pipeline Workflow

Use this skill for end-to-end data ingestion.

## Steps

1. Inspect existing records in PostgreSQL before crawling.
2. Determine which target URLs are new or need refreshing.
3. When browser automation is required, read `references/playwright-cli.md`.
4. Extract page content according to `references/extraction-schema.md`.
5. Validate JSON output with `scripts/validate_json.py`.
6. Insert raw records into PostgreSQL JSONB.
7. Normalize JSONB into relational tables using `scripts/normalize_jsonb.py`.
8. Generate embeddings and store them in pgvector if semantic search is required.

## Rules

- Do not crawl URLs already present unless refresh is requested.
- Always preserve original URL, source title, fetched_at, and raw content.
- Do not invent missing fields; use null or an empty list according to the schema.
- Validate JSON before database insertion.

references/playwright-cli.md 則放「怎麼用 Playwright CLI」：

# Playwright CLI Usage for Crawling

Use Playwright CLI only for pages that require browser rendering, interaction, login-free navigation, or JavaScript execution.

## Basic flow

1. Open the target URL.
2. Inspect visible text and relevant links.
3. Extract the article body or case content.
4. Save screenshots only when visual evidence is needed.
5. Return structured text to the main pipeline.

## Output contract

Return extracted content as JSON-compatible data:

{
  "url": "...",
  "title": "...",
  "published_at": "...",
  "reported_at": "...",
  "case_time": "...",
  "body_text": "...",
  "evidence": [],
  "extraction_notes": []
}

## Guardrails

- Prefer direct HTTP/static parsing first.
- Use Playwright CLI when JS rendering is necessary.
- Avoid infinite scrolling unless the workflow explicitly requests list discovery.
- Keep raw extracted text separate from normalized fields.

更進一步，你可以把 Playwright CLI skill 的內容分成三類：

原本 Playwright skill 內容	放到哪裡
CLI 安裝方式、基本命令	references/playwright-cli.md
常用抓取流程	references/playwright-cli.md
網頁資料抽取 schema	references/extraction-schema.md
重複性操作腳本	scripts/
範例輸出	examples/ 或 references/examples.md
與資料庫互動	主 skill 或 references/postgres-jsonb.md

我不建議把 Playwright CLI 做成獨立 skill 再讓資料管線 skill 去叫它，除非你有很多不同 skill 都會共用瀏覽器操作。例如：

~/.codex/skills/
├── playwright-browser/
├── scam-data-pipeline/
├── ecommerce-data-pipeline/
└── academic-paper-crawler/

這種情況下，獨立 Playwright skill 有價值。但如果你的主要任務就是「資料管線」，那 Playwright 只是內部能力，合併成 references/ 會更穩。

你的場景我會建議這樣分：

scam-case-pipeline/
├── SKILL.md
├── references/
│   ├── playwright-cli.md          # 瀏覽器操作
│   ├── source-discovery.md        # 怎麼找新案例
│   ├── extraction-schema.md       # JSON schema
│   ├── postgres-jsonb.md          # raw JSONB 儲存規則
│   ├── normalization.md           # 正規化規則
│   └── pgvector.md                # embedding / 向量欄位規則
├── scripts/
│   ├── check_existing_urls.py
│   ├── validate_case_json.py
│   ├── insert_raw_jsonb.py
│   ├── normalize_cases.py
│   └── embed_cases.py
└── examples/
    ├── valid_case.json
    └── invalid_case.json

重點是：SKILL.md 放總流程；references 放詳細規則；scripts 放可執行且可重複的步驟。

這比在 skill 裡寫「請讀取 playwright-cli skill」可靠很多，因為 Codex 一旦選中你的資料管線 skill，就能在同一個 skill package 內按需讀取相關文件，不需要賭它是否會正確切換到另一個 skill。
